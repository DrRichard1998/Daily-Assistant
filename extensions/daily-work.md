# daily-work

## 1. 目的

本扩展用于生成 DailyAssistant 每日日报，并支持把日报生成配置成自动化任务。

日报必须基于本项目 SQLite 数据库中的任务、日程和待确认事项生成，内容应简洁、准确、便于快速扫描。生成日报本身只是查询结果视图，不写回业务数据库，也不作为新的事实来源。

## 2. 触发条件

当用户表达以下任一意图时，使用本扩展：

1. 生成今天、当天或指定日期的日报、晨间简报、每日简报。
2. 查看今天的日程、待办、逾期待办、已完成事项和待确认事项的汇总。
3. 创建、配置或调整每天定时生成日报的自动化任务。
4. 输入中包含 `这是一个自动化任务`。

## 3. 入口分流规则

收到输入后，先判断当前请求类型，再进入对应流程。

### 3.1 自动化执行

如果输入中包含字段：

```text
这是一个自动化任务
```

说明当前请求来自自动化程序。此时应：

1. 按本扩展生成执行当天的日报。
2. 将日报写入 `Reports/Daily/YYYY-MM-DD.md`。
3. `YYYY-MM-DD` 使用执行当天的 `Asia/Shanghai` 本地日期。
4. 不询问用户是否写入文件。
5. 不输出多余内容；如需回复，只回复写入结果。

### 3.2 用户直接生成日报

如果输入中不包含 `这是一个自动化任务`，并且用户表达的是“生成今天日报”“输出当前日报”“看看今天日报”等直接生成日报的意图，应：

1. 按本扩展生成日报。
2. 直接在对话中输出日报内容。
3. 不立即写入文件。
4. 输出日报后，询问用户是否需要写入 `Reports/Daily/YYYY-MM-DD.md`。

### 3.3 用户配置自动化任务

如果输入中不包含 `这是一个自动化任务`，并且用户表达的是“每天生成日报”“创建日报自动化”“每天早上提醒/生成简报”等配置自动化任务的意图，应：

1. 询问必要信息，例如执行时间、重复周期、是否使用默认时区 `Asia/Shanghai`。
2. 信息足够后，创建自动化任务。
3. 自动化任务的执行内容写为：

```text
这是一个自动化任务：请按照 extensions/daily-work.md 的要求，生成当日日报并写入文件。只使用 assistant.py query 的返回结果作为事实来源，不要输出多余内容。
```

4. 自动化任务生成后，简要告知用户自动化配置结果。
5. 不把自动化任务本身写入 SQLite 任务或日程，除非用户明确要求记录。

如果当前环境无法真正创建自动化任务，只说明已整理好自动化配置建议，不要假装已经创建。

## 4. 事实来源

唯一事实来源：

1. `data/assistant.sqlite`
2. `assistant.py query` 的返回结果

禁止事项：

1. 不直接查询 SQLite。
2. 不手工写 SQL。
3. 不读取外部 Markdown 文件作为事实来源。
4. 不从历史对话、记忆或推测中补充事项。

## 5. 必要查询

生成当天日报时，优先运行：

```powershell
python .\assistant.py query --period today
```

生成指定日期日报时，运行：

```powershell
python .\assistant.py query --date YYYY-MM-DD
```

如需补充当天已完成任务或日程，可按需运行：

```powershell
python .\assistant.py query --date YYYY-MM-DD --type task --status completed
python .\assistant.py query --date YYYY-MM-DD --type event --status completed
```

如需确保待确认事项为全量活跃队列，运行：

```powershell
python .\assistant.py query --type reviews --status active
```

## 6. 输出目标

自动化执行时，保存为：

```text
Reports/Daily/YYYY-MM-DD.md
```

其中 `YYYY-MM-DD` 为执行当天的 `Asia/Shanghai` 本地日期。

用户直接请求生成日报时，默认只在对话中输出日报内容。除非用户同意或输入中包含 `这是一个自动化任务`，否则不要写入文件。

## 7. 输出格式

```markdown
# 晨间简报｜{{date}}

## 1. 今天结论

{{brief_conclusion}}

## 2. 日程

### 2.1 当天日程

| 时间 | 事项 | 状态 | 备注 |
|---|---|---|---|
| {{start_at}} | {{event_title}} | {{status}} | {{location_or_note}} |

### 2.2 下一条日程

| 时间 | 事项 | 状态 | 备注 |
|---|---|---|---|
| {{start_at}} | {{next_event_title}} | {{status}} | {{location_or_note}} |

### 2.3 当天已完成日程

| 时间 | 事项 | 完成时间 | 备注 |
|---|---|---|---|
| {{start_at}} | {{completed_event_title}} | {{completed_at}} | {{note}} |

## 3. 待办

### 3.1 逾期待办

| 截止时间 | 事项 | 状态 | 备注 |
|---|---|---|---|
| {{due_at}} | {{task_title}} | {{status}} | {{note}} |

### 3.2 当天及之后待办

| 日期 | 截止时间 | 事项 | 状态 | 备注 |
|---|---|---|---|---|
| {{date}} | {{due_at}} | {{task_title}} | {{status}} | {{note}} |

### 3.3 无明确截止时间待办

| 事项 | 状态 | 备注 |
|---|---|---|
| {{task_title}} | {{status}} | 未记录截止时间 |

### 3.4 当天已完成待办

| 截止时间 | 事项 | 完成时间 | 备注 |
|---|---|---|---|
| {{due_at}} | {{completed_task_title}} | {{completed_at}} | {{note}} |

## 4. 待确认事项

| 创建时间 | 问题 | 原因 |
|---|---|---|
| {{created_at}} | {{question}} | {{reason}} |

## 5. 风险提示

{{risk_summary}}
```

## 8. 版块显示规则

所有版块和子版块默认遵循：如果没有可展示条目，则不显示该标题、表格或空状态说明。

具体规则：

1. `今天结论` 始终显示；只总结实际存在的日程、待办、已完成事项、待确认事项和风险。
2. `当天日程` 仅当 `events.in_range` 非空时显示。
3. `下一条日程` 仅当 `events.next_upcoming` 非空时显示。
4. `当天已完成日程` 仅当当天存在 `status: completed` 的日程时显示。
5. `逾期待办` 仅当 `tasks.overdue_before_range` 非空时显示。
6. `当天及之后待办` 仅当 `tasks.due_in_range` 或 `tasks.upcoming_after_range` 非空时显示。
7. `无明确截止时间待办` 仅当 `tasks.without_due_at` 非空时显示。
8. `当天已完成待办` 仅当当天存在已完成任务时显示。
9. `待确认事项` 仅当 `reviews` 非空时显示。
10. `风险提示` 仅当存在明确风险时显示。

不输出“暂无...”类占位文本。如果整份日报没有任何日程、待办、已完成事项、待确认事项或风险，只保留标题，并写一句：

```text
今日暂无已记录事项。
```

## 9. 日程生成规则

1. 日程只来自 `events`。
2. `当天日程` 列出 `events.in_range` 中当天所有日程。
3. `下一条日程` 使用 `events.next_upcoming`。
4. `当天已完成日程` 使用当天 `status: completed` 的日程查询结果。
5. 日程按 `start_at` 升序排列。
6. 任务、截止事项、提醒事项或可以提前完成的事项，即使带有具体时间，也不能放入日程。
7. 如果 `location` 为空，备注写“未记录地点/链接”。
8. 如果日程开始时间早于报告生成时间且仍为 `active`，标为“逾期未完成”。

## 10. 待办生成规则

1. 待办只来自 `tasks`。
2. `逾期待办` 列出 `tasks.overdue_before_range` 中仍未完成的任务。
3. `当天及之后待办` 合并列出 `tasks.due_in_range` 和 `tasks.upcoming_after_range`。
4. `无明确截止时间待办` 列出 `tasks.without_due_at`。
5. `当天已完成待办` 使用当天已完成任务查询结果。
6. 待办按日期分组，同一天内按 `due_at` 升序排列。
7. 不重复列出已经放入 `逾期待办` 的任务。
8. 任务即使有具体时间，也不能放入 `日程`。
9. 若任务原本没有截止时间，截止时间写“未记录截止时间”。

## 11. 待确认事项生成规则

1. 待确认事项只来自 `reviews`。
2. 列出所有 `status: open` 的待确认事项。
3. 必须保留 `question` 原文。
4. 可以展示 `reason`，但不要把原因翻译成数据库中不存在的新事实。
5. 不要把待确认事项改写成确定的任务或日程。

## 12. 状态展示规则

1. `status: active` 的日程写为“待进行”。
2. `status: completed` 的日程写为“已完成”，并展示 `completed_at`。
3. `status: cancelled` 的日程写为“已取消”。
4. `status: active` 的任务写为“未完成”。
5. `status: completed` 的任务写为“已完成”，并展示 `completed_at`。
6. 若日程开始时间早于报告生成时间且仍为 `active`，写为“逾期未完成”。
7. 若任务 `due_at` 早于当天范围且仍为 `active`，放入 `逾期待办`。

## 13. 风险提示规则

仅当存在以下情况时显示 `风险提示`：

1. 当天日程冲突。
2. 已过开始时间但仍为 `active` 的日程。
3. 逾期但仍未完成的待办。
4. 待确认事项可能影响日程或待办记录。
5. 地点、链接、联系人、截止时间等关键信息缺失造成的风险。

风险提示只能基于 CLI 返回字段判断，不要虚构风险。

## 14. 通用生成规则

1. 默认使用 `Asia/Shanghai` 本地日期。
2. 默认中文输出。
3. 严格保持章节顺序：今天结论、日程、待办、待确认事项、风险提示。
4. 不虚构负责人、参与人、地点、链接、优先级、下一步或阻塞项。
5. 不默认输出数据库内部 ID，除非用户明确要求或需要区分同名事项。
6. 生成日报不修改 `data/assistant.sqlite`。
7. 不把自动化任务本身写入 SQLite 任务或日程，除非用户明确要求记录。
